# Ansible for kube_lab

How the Ansible layer configures the lab after Terraform creates VMs and writes inventory.

Related: [overview.md](overview.md) · [terraform.md](terraform.md) · [nfs-storage.md](nfs-storage.md) · [operations.md](operations.md)

## Big picture

Ansible is **layer 2**. It assumes:

1. VMs already exist on Proxmox
2. `ansible/inventory/hosts.proxmox.yml` exists (generated by Terraform)
3. SSH works as `ansible_user` (usually `ubuntu`) with your private key

It then:

1. Prepares the OS (`common`) and routes k3s → lab net (`lab_routes`)
2. Exports NFS on `nfs-01` (`nfs_server`)
3. Installs k3s server on the control plane, agents on workers (`k3s`)
4. Installs the NFS StorageClass via a k3s `HelmChart` (`nfs_provisioner`)
5. Optionally verifies readiness (`verify.yml`)

```text
terraform apply
  → hosts.proxmox.yml
       │
       ▼
site.yml
  1. common+lab_routes → k3s_cluster
  2. common+nfs_server → nfs
  3. k3s (server)    → control_plane
  4. k3s (agent)     → workers
  5. nfs_provisioner → control_plane (if nfs group present)
```

## Directory layout

```text
ansible/
├── ansible.cfg              # defaults: inventory path, become, SSH pipelining
├── requirements.yml         # Galaxy collections
├── inventory/
│   ├── .gitkeep
│   ├── hosts.proxmox.yml    # generated by Terraform (gitignored)
│   └── group_vars/all/      # timezone, packages, kubeconfig path (real dir)
├── playbooks/
│   ├── site.yml             # full bring-up
│   ├── nfs_provisioner.yml  # shared StorageClass play
│   ├── storage.yml          # day-2: import_playbook nfs_provisioner.yml
│   └── verify.yml           # readiness checks
└── roles/
    ├── common/              # OS prep (k8s sysctls optional via flag)
    ├── lab_routes/          # k3s → lab_cidr via Proxmox (NFS reachability)
    ├── k3s/                 # server + agent install (canonical Traefik-off args)
    ├── nfs_server/          # nfs-kernel-server + /etc/exports
    └── nfs_provisioner/     # HelmChart → StorageClass "nfs"
```

There are no `host_vars/`, vault files, or dynamic inventory plugins. Inventory is a static YAML file written by Terraform.

## How to run

Prefer Make (sources `.env`, sets `ANSIBLE_CONFIG` and inventory):

```bash
make ansible    # site.yml
make storage    # storage.yml → import nfs_provisioner.yml
make verify     # verify.yml
```

Ensure `.env` has `ANSIBLE_USER` and `ANSIBLE_PRIVATE_KEY_FILE` matching the cloud-init user/key in tfvars. See [operations.md](operations.md).

Equivalent without Make:

```bash
set -a && source .env && set +a
ANSIBLE_CONFIG=ansible/ansible.cfg \
  ansible-playbook -i ansible/inventory/hosts.proxmox.yml \
  ansible/playbooks/site.yml
```

Install collections once:

```bash
make init
# or: ansible-galaxy collection install -r ansible/requirements.yml
```

### `ansible.cfg` behavior

| Setting | Effect |
|---------|--------|
| `inventory = inventory/hosts.proxmox.yml` | Default inventory when cwd/config is under `ansible/` |
| `become = True` / `sudo` | Root for package and service tasks |
| `host_key_checking = False` | Lab convenience; SSH still uses `accept-new` |
| `pipelining = True` | Faster SSH privilege escalation |
| `stdout_callback = yaml` | Readable task output |

Makefile always passes `-i $(INV)` and `ANSIBLE_CONFIG=...` so runs from the repo root stay consistent.

## Inventory and groups

**Do not edit** `hosts.proxmox.yml` by hand. Change IPs/names in `terraform/terraform.tfvars`, then `make apply`.

### Group hierarchy

```text
all
├── k3s_cluster
│   ├── control_plane     # e.g. k3s-cp-01
│   ├── workers           # e.g. k3s-wk-01, k3s-wk-02
│   └── vars:
│         lab_cidr
│         proxmox_lan_ip
└── nfs                   # sibling of k3s_cluster, not inside it
    ├── hosts: nfs-01
    └── vars:
          nfs_export_path
          nfs_client_cidr
```

### Per-host fields (from Terraform)

| Field | Example | Used for |
|-------|---------|----------|
| `ansible_host` | `192.168.1.20` / `10.10.10.30` | SSH target; k3s API URL; NFS server IP |
| `ansible_user` | `ubuntu` | SSH user (must match tfvars `ssh_user`) |

### Assumptions baked into playbooks

- **One control plane** — token and `K3S_URL` come from `groups['control_plane'][0]`
- **NFS is optional for provisioner** — if `groups['nfs']` is empty, `nfs_provisioner` is skipped
- **NFS hosts are not k3s nodes** — they never get the `k3s` role
- **Play order matters** — control plane must run before workers so `k3s_cluster_token` exists
- **k3s → lab route** — `lab_routes` on `k3s_cluster` needs `lab_cidr` + `proxmox_lan_ip`

### Example generated inventory

```yaml
all:
  children:
    k3s_cluster:
      children:
        control_plane:
          hosts:
            k3s-cp-01:
              ansible_host: 192.168.1.20
              ansible_user: ubuntu
        workers:
          hosts:
            k3s-wk-01:
              ansible_host: 192.168.1.21
              ansible_user: ubuntu

              ansible_host: 192.168.1.22

```

## Group vars loading (important)

Inventory-adjacent vars live at `ansible/inventory/group_vars/all/main.yml` (timezone, packages, kubeconfig path).

Canonical **k3s** install args (`--disable traefik`, version) live in `roles/k3s/defaults/main.yml`. Override via inventory group_vars only when you need different flags.

## Playbooks

### `site.yml` — full bring-up

| Order | Play | Hosts | Roles |
|-------|------|-------|-------|
| 1 | Prepare all k3s cluster nodes | `k3s_cluster` | `common`, `lab_routes` |
| 2 | Configure NFS server | `nfs` | `common` (`common_prepare_kubernetes: false`), `nfs_server` |
| 3 | Bootstrap k3s control plane | `control_plane` | `k3s` (`serial: 1`) |
| 4 | Join k3s workers | `workers` | `k3s` |
| 5 | `import_playbook: nfs_provisioner.yml` | `control_plane` | `nfs_provisioner` (if `nfs` group non-empty) |

No tags. Subset runs use separate playbooks (`storage.yml`) or `--limit`.

### `verify.yml` — readiness

Runs on `control_plane` (`gather_facts: false`):

1. Wait until `kubectl get nodes` shows every host in `k3s_cluster` as `Ready`
2. If `nfs` group present: `showmount -e` from the CP and assert `nfs_export_path` is exported
3. If `nfs` group present: `kubectl get storageclass nfs` succeeds

Does **not** create a PVC (kept non-mutating). Manual bind test:

```bash
export KUBECONFIG="$PWD/kubeconfig"
kubectl apply -f manifests/storage/pvc-smoke.yaml
kubectl get pvc nfs-smoke
kubectl delete -f manifests/storage/pvc-smoke.yaml
```

## Roles

### `common` — base OS for the lab

**Applied to:** all of `k3s_cluster`, and again on `nfs` (with `nfs_server`).

**Always (all hosts that get this role):**

1. Set `timezone` (`America/Chicago` by default in inventory group_vars)
2. Wait for apt locks (`fuser` on dpkg lock)
3. `apt` update (`cache_valid_time: 3600`)
4. Install `common_packages`
5. Enable `qemu-guest-agent` (`failed_when: false` so a missing unit does not fail the play)

**Only when `common_prepare_kubernetes` is true** (default; k3s nodes):

6. Load and persist `br_netfilter`
7. Sysctl: bridge netfilter + `ip_forward`
8. Disable swap now and comment swap lines in `/etc/fstab`

NFS play sets `common_prepare_kubernetes: false`, so the NFS VM only gets packages/timezone/guest-agent.

**Default packages** (inventory `group_vars`):

`curl`, `ca-certificates`, `open-iscsi`, `nfs-common`, `jq`, `qemu-guest-agent`

`nfs-common` on the control plane is what lets `verify.yml` run `showmount`.

---

### `lab_routes` — reach NFS on vmbr1

**Applied to:** `k3s_cluster` only (after `common`).

**Required from inventory** (`k3s_cluster` vars from Terraform): `lab_cidr`, `proxmox_lan_ip`.

**Steps:**

1. Assert those vars are set
2. Write `/etc/netplan/60-kube-lab-route.yaml` (match `en*`, route `lab_cidr` via `proxmox_lan_ip`)
3. `netplan apply` when changed
4. `ip route replace` so the route is live immediately

Without this, PVC mounts and `showmount` from k3s to `10.10.10.30` fail even when PVE forwarding is correct. See [lab-network.md](lab-network.md).

---

### `k3s` — install server or agent

**Defaults** (`roles/k3s/defaults/main.yml`) are canonical for version and server args:

```yaml
k3s_server_args: >-
  --disable traefik
  --write-kubeconfig-mode 644
```

**Dispatch** (`tasks/main.yml`):

- Host in `control_plane` → `server.yml`
- Host in `workers` → `agent.yml`

#### Server (`server.yml`)

1. Skip install if `/usr/local/bin/k3s` already exists (`creates:` / `stat`) — **create-once**; changing `k3s_version` / args requires removing the binary (or rebuilding VMs) then re-running Ansible
2. Otherwise: `curl get.k3s.io | INSTALL_K3S_VERSION=… sh -s - server …`
3. Ensure `k3s` service enabled/started
4. Wait for node-token; slurp it; set `k3s_cluster_token`
5. `delegate_facts` that token to every host in `k3s_cluster` (so workers can join)
6. Fetch `/etc/rancher/k3s/k3s.yaml` to `kubeconfig_local_path` (repo root `kubeconfig`)
7. On localhost, rewrite `https://127.0.0.1:6443` → `https://{{ ansible_host }}:6443`

#### Agent (`agent.yml`)

1. Assert `k3s_cluster_token` is set
2. Build `k3s_server_url` from the first control plane’s `ansible_host`
3. Install agent if binary missing (`K3S_URL` / `K3S_TOKEN`; `no_log: true`)
4. Ensure `k3s-agent` enabled/started

**Idempotency caveat:** install is gated on the binary existing. Re-running Ansible will **not** upgrade k3s or change install-time args if the binary is already there. To change version/args, remove the binary/service on the node or rebuild the VM.

---

### `nfs_server` — export for the cluster

**Required from inventory** (Terraform → `nfs` group vars): `nfs_export_path`, `nfs_client_cidr`. The role **asserts** these are set — there is no path/CIDR fallback in role defaults.

**Role defaults only:** `nfs_export_options`, `nfs_packages`.

**Steps:**

1. Assert inventory provides path + CIDR
2. Install `nfs-kernel-server`
3. Create export directory mode `0777` (provisioner runs as root in-cluster; pairs with `no_root_squash`)
4. Write `/etc/exports` for `nfs_client_cidr` with `rw,sync,no_subtree_check,no_root_squash`
5. Enable/start `nfs-server`
6. On exports change only: handler runs `exportfs -ra`

Export is limited to the lab CIDR — not world-readable.

---

### `nfs_provisioner` — dynamic PVs

Runs on the **control plane** (needs `kubectl` / local k3s kubeconfig).

**Steps:**

1. Resolve `nfs_server_ip` and `nfs_export_path` from `groups['nfs'][0]` hostvars
2. Template `templates/helmchart.yaml.j2` to `/tmp/nfs-provisioner-helmchart.yaml`
3. `kubectl apply -f` that file
4. Wait until StorageClass `nfs` exists

The template creates:

- Namespace `nfs-provisioner`
- k3s `HelmChart` (`helm.cattle.io/v1`) installing **nfs-subdir-external-provisioner**
- StorageClass name `nfs`, reclaim `Retain`, access mode RWX, **not** the default class

Canonical path is this role (`make ansible` / `make storage`). See `manifests/storage/README.md` for the manual PVC smoke test only.

## Variables reference

### From `inventory/group_vars/all/main.yml`

| Variable | Default / purpose |
|----------|-------------------|
| `timezone` | `America/Chicago` |
| `common_packages` | curl, ca-certificates, open-iscsi, nfs-common, jq, qemu-guest-agent |
| `kubeconfig_local_path` | `{{ playbook_dir }}/../../kubeconfig` |

### From Terraform → inventory

| Variable | Where | Consumers |
|----------|--------|-----------|
| `ansible_host` / `ansible_user` | each host | SSH, API URL, NFS IP |
| `nfs_export_path` | `nfs` group vars | `nfs_server`, provisioner, verify |
| `nfs_client_cidr` | `nfs` group vars | `nfs_server` |

### Role defaults (selected)

| Role | Variables |
|------|-----------|
| `k3s` | `k3s_version`, `k3s_server_args` (Traefik off), install URL, token/kubeconfig paths |
| `common` | `common_prepare_kubernetes` (default true) |
| `nfs_server` | `nfs_export_options`, `nfs_packages` (path/CIDR from inventory) |
| `nfs_provisioner` | namespace, StorageClass name/default/reclaim, Helm chart repo/name |

### Runtime facts

| Fact | Set by | Used by |
|------|--------|---------|
| `k3s_cluster_token` | control plane `server.yml` | workers `agent.yml` |
| `k3s_server_url` | agent tasks | install command |
| `nfs_server_ip` | provisioner / verify | HelmChart + showmount |

**Precedence that matters:** inventory group vars beat role defaults for shared names. Traefik-off is in **role defaults**, so it applies even if group_vars are thin.

## Terraform ↔ Ansible contract

| Concern | Terraform | Ansible |
|---------|-----------|---------|
| Hostnames, IPs, SSH user | VMs + inventory hosts | Connection + URLs |
| NFS path / client CIDR | tfvars → `nfs` group vars | `nfs_server` + provisioner |
| `cluster_name` | Proxmox VM tags only | not exported to Ansible inventory |
| Guest agent package | VM agent flag off in provider | `common` enables the service |
| k3s / NFS packages / HelmChart | — | Ansible only |
| kubeconfig on laptop | — | fetched + rewritten by `k3s` |

Ansible does **not** read Terraform state. It only reads the generated inventory (and inventory `group_vars`).

| Entry point | Ansible involvement |
|-------------|---------------------|

| `make ansible` | `site.yml` |

## Day-2 recipes

| Goal | Commands |
|------|----------|
| Reconfigure OS / reinstall path after VM recreate | `make ansible` |
| Only fix / refresh NFS StorageClass | `make storage` |
| Confirm nodes + export + SC | `make verify` |
| PVC smoke test | `kubectl apply -f manifests/storage/pvc-smoke.yaml` |
| Change NFS path or CIDR | Update tfvars → `make apply` → `make ansible` |
| Change / upgrade k3s | Update `roles/k3s/defaults` → remove `/usr/local/bin/k3s` on nodes (or rebuild VMs) → `make ansible` (install is create-once) |
| Add a worker | Add worker in tfvars → `make apply` → `make ansible` (or limit to new host after CP has run once) |

Day-2 patterns and troubleshooting live in [operations.md](operations.md). Ansible-focused failure modes:

| Symptom | What to check |
|---------|----------------|
| Inventory not found | Run Terraform apply first; file is gitignored |
| Unreachable hosts | SSH keys (`ANSIBLE_PRIVATE_KEY_FILE`), `ansible_user`, cloud-init finished |
| Traefik still present | Confirm `roles/k3s/defaults` args; reinstall if binary already existed without `--disable traefik` |
| Workers fail token assert | Control plane play must succeed first; do not run workers alone on a fresh cluster |
| `showmount` fails | `nfs_server` role, export CIDR includes CP, `nfs-common` on CP |
| StorageClass never appears | HelmChart apply, NFS IP wrong, k3s Helm controller, `kubectl -n kube-system get helmchart` |
| PVC Pending | Provisioner pods, NFS mount from nodes, reclaim leftovers on disk |

## Security notes (lab posture)

- NFS root is mode `0777` with `no_root_squash` so the in-cluster provisioner can create PVC directories — convenient for a lab, not a hardened multi-tenant design
- Export is scoped to `nfs_client_cidr`
- `host_key_checking` is off in `ansible.cfg` for rebuild convenience
- Cluster token is shared via Ansible facts in-memory during the play (not written to group_vars)
